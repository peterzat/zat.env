## Spec — 2026-05-01 — /tester design D.4/D.5.5/D.6 ordering fix

**Goal:** Restore Step D.5.5 (the pre-apply checklist shipped earlier today) as a TRUE pre-mutation gate by removing the file-write semantic from Step D.4. Currently D.4 writes the contract to TESTING.md before D.5.5 fires, so the checklist is post-mutation for TESTING.md and the LLM correctly recognizes it as redundant with D.7's report — collapsing it. The fix makes D.4 a draft-in-memory step, has D.5.5 reference the draft, and consolidates the actual write into D.6 step 1. After this turn, D.5.5 is the unambiguous "stop and let the user veto before any TESTING.md or BACKLOG.md mutation" gate.

### Acceptance Criteria

- [x] **D.4 has draft-in-memory semantics.** The Step D.4 section in `claude/skills/tester/SKILL.md` is renamed and reworded so the LLM understands its job is to compose the contract content, not to write to TESTING.md. The opening prose names this explicitly (e.g., "Draft the contract section in memory; no file write happens here — Step D.6 owns the write"). The proportionality rules (greenfield ~50 lines soft cap, growing two-tier, mature full-dimension) and the 6-section minimum structure (cold-open, entry point, duration philosophy, proxy/drift, human-eval, what-not-to-test) stay intact in D.4 — they are drafting guidance, not write instructions. The phrase "at the bottom of TESTING.md" with imperative-write framing is removed.

- [x] **D.4 no longer carries the revision-behavior text.** The paragraph beginning "Revision behavior: if a prior `# Durable test-architecture contract` H1 exists in TESTING.md, replace everything from that H1 through the end of the file with the new contract" is removed from D.4 (where the write semantic justified its presence). It moves to D.6 step 1, where the actual write happens.

- [x] **D.5.5 references the in-memory draft, not a written file.** D.5.5's "contract shape + line count" component prose updates to make clear the line count is from the LLM's drafted contract (not from TESTING.md on disk). The earlier phrasing "the contract section just written to TESTING.md" or equivalent post-write framing is removed. The 5 components and the always-on / flag-not-block guards stay intact (existing lint already enforces these).

Content above the H1 stays intact. If no prior contract existed, append the H1 block (or create TESTING.md containing only the H1 + body if the file did not exist)." D.6 step 2 (BACKLOG manifest via `spec-backlog-apply.sh`) is unchanged.

The existing D.5.5 anchor / position / 5-component checks continue to pass without modification.

### Context

**Why this turn.** External audit of the prior turn's hardening identified a real structural ambiguity that explains the D.5.5 compliance failure observed on two PanelForge runs (the new pre-apply checklist did not fire as a structured 5-component block; the LLM collapsed it into D.7's post-mortem report). Re-reading the SKILL prose: D.4 says "Write the contract under the EXACT H1 heading `# Durable test-architecture contract` at the bottom of TESTING.md" — that's a write-to-disk semantic, so TESTING.md is mutated by the time D.5.5 fires. D.5.5 then references "the line count of the contract section just written to TESTING.md," which only makes sense AFTER a write. D.6 step 1 then says "Write the contract to TESTING.md" again. From the LLM's reading, D.5.5 has the same epistemic position as D.7 (both post-write commentary on already-mutated state), so the LLM correctly recognizes the redundancy and skips D.5.5. The hardening turn shipped a feature that's structurally dormant.

**Design choices made (alternatives on request).**

- **D.4 as draft, D.6 as write.** Alternative: keep D.4 as the write step and remove D.6 step 1 entirely. Chose draft-then-write because it makes D.5.5 a natural pre-mutation gate (no mutations between D.4 and D.5.5; the user can interrupt without anything to revert). Removing D.6 step 1 would still leave D.5.5 as a recap rather than a gate. The fix is structural, not cosmetic.
- **Move revision-behavior text to D.6, not duplicate it.** Alternative: leave the H1-replace text in D.4 and also reference it in D.6. Chose move because duplicate prose drifts; the revision behavior is a write-time concern, so it belongs at the write site.
- **Lint enforces structure, not behavior.** Alternative: add a behavioral test that spawns a subagent and greps its output for the 5-component pattern. Chose to defer the behavioral test (out of scope for this turn) because it's real engineering work; the structural lint is the cheap fix that catches drift, and PanelForge re-run is the manual behavioral validation.
- **No update to D.5.5's 5-component spec.** The components stay (signals fingerprint, contract shape + line count, rollout count + justification, per-entry overlap scan, optional SPEC tension); only the prose framing of "from your draft" vs. "from the written file" shifts. The user's experience of the checklist is unchanged in shape.

**Validation path.** Lint and behavioral tests verify structural compliance. Behavioral compliance (does the LLM actually post the 5-component block on a real `/tester design` run?) is validated by manually re-running `/tester design` on PanelForge after this turn lands. Success: the agent posts a structured 5-component block before any TESTING.md or BACKLOG.md mutation, and `git diff TESTING.md` shows no change at the point the checklist appears in the agent's output. If the checklist still doesn't fire after this fix, the issue is deeper than ordering and warrants investigation beyond prose tuning.
