## Spec — 2026-05-01 — /tester design hardening from PanelForge feedback

**Goal:** Add a visible pre-apply checklist to `/tester design` so the user can course-correct before the script mutates BACKLOG.md, plus four smaller refinements (overlap scan strengthening, overlap-append handling, line-cap softening, boilerplate-Why-deferred soft hint) drawn from a real greenfield run on PanelForge. The hardening targets the only "evaluate what you just did" surface the skill currently lacks; everything else is incremental polish on the same skill.

### Acceptance Criteria

- [x] **Step D.5.5 exists.** `claude/skills/tester/SKILL.md` contains a new section titled `### Step D.5.5: Pre-apply checklist (visible to user)`, positioned between the existing Step D.5 and Step D.6. The section instructs the LLM to post a fixed-structure block to the user before the script invocation in Step D.6, and explicitly enumerates the five required components below.

- [x] **Pre-apply checklist structure.** The block has these components, posted in order: (a) Signals fingerprint — one line drawn from Step D.2. (b) Contract shape chosen (greenfield seed / growing two-tier / mature full-dimension) plus the line count of the contract section just written to TESTING.md. (c) Rollout entry count plus a one-line justification. (d) Per-entry overlap scan result — one line per rollout entry, of the form `<short name>: no overlap` or `<short name>: overlaps <other-entry-name>`; this component is omitted only when there are zero rollout entries. The skill prompt names each component so the lint check can verify the enumeration.

- [x] **SPEC tension is flag, not block.** When component (e) fires, the skill posts the tension line and proceeds directly to Step D.6 — no user-confirmation round-trip, no halt. The skill prompt states this in plain prose so a future revision does not silently convert the flag into a gate.

Topical overlap is judgment-based — substantively the same goal or strongly overlapping problem area — not name-match. Scan results are the source for the per-entry overlap line in Step D.5.5.

- [x] **Overlap-append handling.** When Step D.1's scan plus Step D.5's drafting identifies a topical overlap with a non-tester BACKLOG entry, the rollout entry's body (written verbatim inside the `append:` / `end-append` block in Step D.6) gains a `**Coordinate with:** <other-entry-name>` line appended after the `Origin:` field. ### Context

**Why this turn.** The prior turn (2026-04-24) shipped `/tester design` mode with audit/design dispatch, the durable contract H1, rollout entries via `spec-backlog-apply.sh`'s new `purge-origin` and `append` ops, and full lint coverage. PanelForge (`~/src/PanelForge`, single-file HTML demo, no test framework) was the first real-world greenfield run. The output was solid — 52-line greenfield seed in TESTING.md, 6 well-formed rollout entries in BACKLOG.md (one of which manually noted overlap with an existing entry by referencing it in body prose) — but the run had no surface for the user to evaluate the LLM's proportionality and overlap judgments before BACKLOG.md mutated. Specifically: the proportionality call (52 lines vs. 50-line cap; 6 entries at the upper bound of the 3–6 range) was made silently in Step D.3 ("out loud to yourself; not to the user"), and a SPEC tension worth flagging (PanelForge SPEC.md explicitly puts automated tests out of scope this turn while the rollout proposes adding them) went unreported. This turn closes those gaps.

**Design choices made (alternatives on request).**

- **Always-on visible checklist, no flag-gating.** Alternative: gate behind `--show-reasoning` or similar. Chose always-on because the checklist is the only window the user has into the proportionality and overlap calls; making it opt-in defeats the purpose for the user who would benefit most.
- **Flag-not-block on SPEC tension.** Alternative: halt and require user confirmation. Chose flag-not-block because the rollout-as-latent-backlog use case is legitimate (entries flow through `/spec`'s revisit-criteria loop later); halting forces friction on the common case.
- **Append-with-coordinate-line on overlap, not skip.** Alternative: skip the overlap entry, or surface and let the user choose at runtime. Chose append-with-coordinate-line because PanelForge already did this manually (`real-mode-canary-when-key-available` referenced `openrouter-model-availability-check` in body prose) and it is the right shape — the rollout is real work the user wants tracked, and the coordinate field gives `/spec`'s adoption flow visibility into the relationship.
- **Soft hint on boilerplate Why-deferred.** Alternative: hard check that fails the run if N of M entries use the boilerplate phrase. Chose soft hint because hard checks tend to push the LLM into over-explaining; the goal is judgment, not enforcement.
- **No script changes.** All hardening is prompt-only or lint-only. `bin/spec-backlog-apply.sh` is unchanged; the `append:` / `end-append` block already preserves arbitrary body content, including the `Coordinate with:` field.

**Files in scope.** `claude/skills/tester/SKILL.md` (new D.5.5; D.1, D.4, D.5 edits), `tests/lint-skills.sh` (new checks in the Tester/spec section), `CLAUDE.md` (one new contract-point bullet), `README.md` (one new bullet in Design-mode flows). "Prompts must earn their keep": D.5.5 names the failure mode it prevents (silent proportionality and overlap calls). Writing-style: professional, direct, concise; no AI-voice patterns; no em-dashes in stored prose.
