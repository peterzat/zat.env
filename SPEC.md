## Spec — 2026-05-01 — Deterministic codereview marker write/verify script

**Goal:** Eliminate the LLM-split-Bash-call failure mode in codereview's marker write (and the related UPSTREAM derivations in Steps 2 and 5.5) by extracting all UPSTREAM / PROJ_HASH / DIFF_HASH computation into a deterministic shared `bin/` script that codereview and the pre-push hook both invoke. Replaces the current parallel-bash-snippets contract (where the codereview skill prose and the hook script must stay byte-for-byte identical) with a single shared implementation no prose-vs-prose drift can break.

### Acceptance Criteria

- [x] **Marker script exists with three-case upstream contract.** A new `bin/` shell script (one dual-mode script or a write/verify pair — implementation choice) computes the codereview push marker hash from the project's working-tree state. The script handles three upstream cases consistently: (a) `@{upstream}` resolves — diff against that ref; (b) `@{upstream}` does not resolve but `origin/<current-branch>` does — diff against `origin/<current-branch>` (the PanelForge case the prior bash snippet handled correctly only when run as one Bash invocation); (c) neither resolves — diff against the empty tree (first push of a new branch). The script also encapsulates `PROJ_HASH` derivation from the repo root (currently duplicated across `bin/codereview-skip`, `hooks/pre-push-codereview.sh`, and codereview SKILL.md Step 8) and the marker file path template. Uses `set -euo pipefail`, idempotent, shellcheck clean at warning level.

- [x] **Codereview Step 8 invokes the script.** `claude/skills/codereview/SKILL.md` Step 8 (Write Marker File) replaces its current bash snippet with a single invocation of the new script. The skill prompt contains zero `UPSTREAM=$(git rev-parse --abbrev-ref '@{upstream}'` patterns, zero inline `if git rev-parse "${UPSTREAM}"` branching, zero inline `git diff "${UPSTREAM}" ... | sha256sum` computation. Step 8 reduces to a one-line script call (plus the existing "Do NOT write the marker if any BLOCK items remain" guard).

- [x] **Pre-push hook invokes the script.** `hooks/pre-push-codereview.sh` replaces its inline UPSTREAM derivation, BASE selection, and DIFF_HASH computation with a call to the same script (verify mode, or peer verify script). The hash-comparison logic in the hook (marker file present, hash matches, BLOCK guard) is unchanged; only the hash *computation* moves into the script. The hook's single-shell-process correctness is preserved.

- [x] **Steps 2 and 5.5 hardened against shell-state-loss.** `claude/skills/codereview/SKILL.md` Step 2 (refresh detection) and Step 5.5 (external reviewers) currently each derive `UPSTREAM` and reference it later in the same bash block. Both sites are restructured so the LLM-split-Bash-call failure mode cannot apply: either by invoking the new script for any UPSTREAM-dependent computation, or by collapsing the derivation and use into a single self-contained one-line pattern (e.g., `git diff "$(git rev-parse --abbrev-ref '@{upstream}' 2>/dev/null || echo "origin/$(git rev-parse --abbrev-ref HEAD)")" -- ...`). The acceptance check is structural: no multi-statement bash block in the SKILL.md sets `UPSTREAM=` and references `${UPSTREAM}` on a later line.

- [x] **Bare-name PATH invocation.** Codereview SKILL.md and the pre-push hook invoke the new script(s) by bare name (no `bin/` prefix). The script(s) are symlinked into `~/bin/` by `zat.env-install.sh` alongside `spec-backlog-apply.sh` and `codereview-skip`. The same lint pattern that protects `spec-backlog-apply.sh` from `bin/` prefix drift covers the new script(s).

- [x] **Behavioral test coverage for the three upstream cases plus empty diff.** A new `tests/test-codereview-marker.sh` (or extension of `tests/test-pre-push-hook.sh`) exercises four scenarios in isolated `mktemp`-d git repos: (a) `@{upstream}` set and resolves; (b) `@{upstream}` unset but `origin/<branch>` resolves (the PanelForge case, named explicitly in the test description); (c) neither resolves; (d) only excluded review files differ from upstream (empty effective diff). Each scenario asserts the script's hash output is stable across two invocations and matches what the hook would compute for the same state. The new suite is wired into `tests/run-all.sh`.

- [x] **Lint coverage in `tests/lint-skills.sh`.** New checks under a Codereview marker contracts section (or extension of the existing Marker file gating section): (a) `has` check that `claude/skills/codereview/SKILL.md` Step 8 contains the script invocation by bare name. (b) `hasnt` checks that Step 8 does not contain the inline patterns it replaced (`UPSTREAM=$(git rev-parse --abbrev-ref`, `if git rev-parse "${UPSTREAM}"`, `git diff "${UPSTREAM}" ... | sha256sum`). (c) `has` check that the pre-push hook contains the script invocation. (d) `hasnt` check that the SKILL.md does not invoke the new script with a `bin/` prefix. (e) The shellcheck loop already covers any new script in `bin/` automatically.

- [x] **CLAUDE.md contract bullet updated.** The `Marker hash computation` bullet under Prompt/infrastructure boundary in `CLAUDE.md` is rewritten to reflect that hash computation is now performed by a shared `bin/` script invoked by both codereview and the hook (single source of truth) rather than parallel bash snippets that must stay byte-identical. The bullet names the script and explains the failure mode that motivated the change (LLM-split-Bash-call shell-state-loss surfaced in PanelForge).

- [x] **No regressions, marker round-trips end-to-end.** `tests/run-all.sh` succeeds with all prior 456 checks still passing (or intentionally updated where this turn invalidates a prior anchor) plus the new behavioral tests from criterion 6 and lint checks from criterion 7. End-to-end smoke verification: a manual `/codereview` run on a branch lacking `@{upstream}` (the PanelForge condition) produces a marker hash that matches what the pre-push hook computes for the same state.

### Context

**Why this turn, why now.** The marker-write extraction was identified as the "highest-value next improvement" in the 2026-04-09 spec's proposal section, then deferred for `/tester design` (shipped 2026-04-24) and the design-mode hardening (shipped 2026-05-01 earlier today). The bug surfaced again in PanelForge after `/tester design` landed: the codereview skill silently wrote an empty-tree marker hash on a branch lacking `@{upstream}` tracking, while the pre-push hook computed the correct `origin/<branch>` hash from the same state, and the push was blocked silently. The hash divergence is purely a shell-state-loss artifact: codereview and the hook share byte-for-byte identical UPSTREAM derivations on paper, but Bash tool calls do not preserve shell variables across invocations, so when the LLM splits the Step 8 snippet for readability, `${UPSTREAM}` is empty in the second call and the else-branch (empty-tree fallback) runs.

**Design choices made (alternatives on request).**

- **Move UPSTREAM derivation into the script, not just the hash computation.** Alternative: keep `UPSTREAM=` in the SKILL prompt and pass it as an argument to the script. Chose to encapsulate UPSTREAM derivation entirely because the prompt contract is what's vulnerable to LLM splitting; reducing the prompt to a single bare-name script call eliminates the failure mode at the source.
- **Implementation choice (single dual-mode script vs. write/verify pair).** Alternative: prescribe one or the other in the criteria. Left as implementer's choice because the contract is behavioral (codereview writes, hook verifies, both compute the same hash for the same state); the file layout doesn't matter to correctness.
- **Harden Steps 2 and 5.5 in the same turn.** Alternative: only fix Step 8 (the observed failure) and defer 2 and 5.5. Chose to harden all three because they have the identical vulnerability pattern; fixing one and leaving the others is asking the same bug to surface in a different code path. The marginal cost of restructuring 2 and 5.5 alongside 8 is small.
- **Self-contained one-liner is acceptable for Steps 2 and 5.5.** Alternative: require Steps 2 and 5.5 to use the new script too. Allowed self-contained one-liners because they're the simplest fix when the snippet only needs UPSTREAM once and immediately; the script call is overkill for those sites. The criteria is structural (no multi-statement UPSTREAM= ... ${UPSTREAM} pattern), not prescriptive about which fix is used.
- **No changes to other skills.** Alternative: extend the script to /security and /tester audit too. Out of scope; this turn is the codereview-specific UPSTREAM contract. Other skills can adopt the script in a future turn if they hit the same failure mode.

**Files affected (rough enumeration; the implementer refines).** `bin/<new-script(s)>` (the work), `claude/skills/codereview/SKILL.md` (Step 8 replaced; Steps 2 and 5.5 restructured), `hooks/pre-push-codereview.sh` (hash computation moves to script call), `tests/lint-skills.sh` (new lint checks; existing Marker file gating section may need updating), `tests/test-codereview-marker.sh` (new) or `tests/test-pre-push-hook.sh` (extended), `tests/run-all.sh` (wires new suite if added), `zat.env-install.sh` (symlinks new script(s) into `~/bin/`), `CLAUDE.md` (Marker hash computation bullet rewritten).

**Out of scope this turn.** Changes to /security, /tester audit, /spec, or other skills' bash snippets. The shell-state-loss vulnerability pattern likely exists elsewhere; tackle when observed, not preemptively.

**Relevant zat.env practices.** Shell scripts use `set -euo pipefail`, are idempotent, guard with existence checks, and stay shellcheck-clean at warning level (existing scripts and lint enforce this). Prompt/infrastructure boundary: this turn is the model case for the boundary — the failure was a prose-prose contract (codereview SKILL.md prose vs. hook script prose) drifting silently because of an LLM-execution artifact. The fix replaces both sites with a script call, eliminating the boundary entirely. Coding practices: small committable increments — script lands first with its tests, then SKILL.md edits, then hook edits, then lint, then docs, with `tests/run-all.sh` between each. Writing-style: professional, direct, concise.
